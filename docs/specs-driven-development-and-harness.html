Specs-Driven Development + Harness

A delivery model where every feature moves through explicit phases, every assumption becomes a tracked question, and every change is verified by a deterministic quality harness before code review.

No invention in phases 1-3 Epic mode: 3a -> 3b planning TDD red -> green -> refactor 10-layer validation harness Full AC to test traceability

What is specs-driven development?

Specs-driven development treats written artifacts as the execution contract between humans and agents. Instead of jumping directly to code, a feature goes through seven phases with clear entry and exit gates.

  1. Specify: write 01-spec.md with ACs and open questions.
  2. Spec review: verify clarity and testability in 02-spec-review.md.
  3. Plan: normal work → 03-design.md + 04-tasks.md; large epic work → 03-epic-design.md + 03a-epic-roadmap.md first, then slice-level design and tasks.
  4. Implement: execute strict TDD with 05-implementation-log.md.
  5. Test: add cross-cutting tests in 06-test-plan.md.
  6. Validate: run the harness and publish reports in 07-validation-report.md.
  7. Code review: publish findings in 08-code-review.md.

What is the harness?

The harness is a deterministic pipeline that validates the work the same way in local runs and CI. It prevents "looks good" decisions and replaces them with measurable gate outcomes.

  • Same script for every environment: harness.sh
  • Layered checks from formatting to security
  • Machine-readable outputs for automated report parsing
  • Baseline-aware for brownfield adoption

How Epic planning works

For large initiatives, planning is split into two levels so teams avoid committing to detailed tasks too early.

  1. Phase 3a (Epic level): write 03-epic-design.md for boundaries and shared decisions, and 03a-epic-roadmap.md for vertical slice sequencing.
  2. Phase 3b (Slice level): run detailed slice design and task breakdown in 03-design.md and 04-tasks.md.
  3. Gate: do not finalize detailed tasks until Epic-level open questions are resolved (or explicitly deferred with rationale).

Operationally, this maps to /epic-plan first, then /plan for slice-level decomposition.

Why document every step?

Without documentation With documentation artifacts Outcome
Hidden assumptions and silent defaults Open questions tracked as Q-NNN and resolved explicitly Fewer requirement surprises
Tasks become broad and ambiguous Task-level scope, AC links, test IDs, and file boundaries Smaller, safer delivery units
"Tested" means different things to different people Shared harness layers and deterministic gate reports Consistent quality bar
Hard to explain why a change exists AC -> task -> test -> code traceability matrix Fast audits and easier maintenance
Review relies on memory and intuition Structured review artifact with blocker/major/minor findings Higher signal in PR discussions

10 harness layers (quick view)

  1. Format and lint
  2. Compile
  3. Static analysis
  4. Architecture rules
  5. Unit and slice tests
  6. Integration tests
  7. Coverage policy
  8. Mutation testing
  9. API contract diff
  10. Dependency security scan

If one layer fails, the feature is not "done" yet. The harness is the feedback loop, not an optional checklist.

Examples

Example 1: Missing pagination rule

During specification, no pagination limit is defined for a list endpoint. Instead of inventing one, the author adds Q-003: "What is the default and max page size?".

Result: the decision is explicit before implementation, and tests can assert exact behavior.

Example 2: Red before green

For task T-002, a failing test is first captured in .tdd-state.json with a failure excerpt. Only then does production code editing proceed.

Result: no accidental code-first drift; every behavior change has a proving test.

Example 3: Validation catches regression

Integration tests pass, but OpenAPI diff reports a breaking response change. The harness blocks the phase until contract compatibility is restored or formally approved.

Result: downstream clients do not get surprised in production.

Example 4: Epic sliced before task breakdown

A feature spans checkout, billing, and receipt flows. Instead of writing all tasks at once, the team creates 03-epic-design.md and 03a-epic-roadmap.md, then plans and implements one vertical slice at a time.

Result: lower rework, clearer dependencies, and faster feedback per slice.

When this approach applies (and when it does not)

Use specs-driven + harness for

  • New features with multiple business rules and edge cases
  • API changes where compatibility and contract safety matter
  • Schema and migration work that can impact production data
  • Security-sensitive flows (auth, permissions, secrets, PII)
  • Cross-service integrations and reliability-critical behavior
  • Changes that require auditability and explicit decision history

Do not use the full process for

  • Typos, copy updates, and tiny cosmetic edits
  • Formatting-only or file-rename housekeeping work
  • Throwaway prototypes that will not be shipped
  • Very small internal refactors with no behavior change
  • Immediate incident mitigation where speed is the first objective

Rule of thumb: if the change can break user behavior, contracts, security, or data, run the full workflow. If the change is low-risk and easily reversible, use a lighter path with focused tests and validation.

Bottom line

Specs-driven development is not "more documents" for the sake of process. It is a risk-control system: decide explicitly, implement incrementally, validate mechanically, and review with evidence.

When teams document each step, they trade hidden complexity for visible decisions. That is what makes velocity sustainable.
